# HTTP basics for TAM debugging  ## Request vs Response - Request = method + URL/path + headers + body - Response = status + headers + body  ## Headers I check first - Host = target domain (and optional port) for the request - Authorization = credentials (Bearer token / API key) - Content-Type = media type of the request body (e.g., application/json) - Accept = expected media type of the response - Idempotency-Key = prevents duplicate processing (important in fintech)  ## Status codes: meaning + first action - 200 OK = succeeded; response usually contains result - 201 Created = resource created; may include Location header - 202 Accepted = accepted for async processing; expect job/status tracking - 204 No Content = succeeded; no body (don't parse JSON) - 400 Bad Request = malformed request (syntax/params); check method/params, headers, JSON validity - 401 Unauthorized = auth failed (missing/invalid/expired credentials); check Authorization, environment, token expiry/issuer - 403 Forbidden = authenticated but not allowed; check scopes/roles/permissions - 404 Not Found = wrong endpoint/id; confirm path, resource id, environment - 409 Conflict = state/idempotency conflict; check duplicates, resource state, idempotency key - 422 Unprocessable Content = valid format, invalid business rules; example: refund amount exceeds captured amount - 429 Too Many Requests = rate limited; backoff + jitter, reduce concurrency/throughput, respect Retry-After - 500/502/503 = server/proxy errors; check provider status/logs and retry carefully  ## Fast triage: what I ask for (customer/provider) - Exact endpoint + method + query params - Timestamp + timezone - Full status code + response body (redact secrets) - Request headers (redact secrets) - Request ID / correlation ID (if any) - Rate limit policy + Retry-After/X-RateLimit-* (if 429)
